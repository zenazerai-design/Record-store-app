Strategic pull

Agent Octopus—and a bet on self-serve

The business aimed to ship a new agentic product—Agent Octopus—with a strong emphasis on self-serve. That timetable made dependable user lifecycle, access, and visibility non-negotiable: not parallel “admin projects,” but part of the same roadmap as API-first and agent workflows.

I treated that launch as leverage to push user management toward something built for where GWI is going, not only where it started. Alongside engineers, the API/MCP team, and designers on the public product, we aligned on what the alpha could prove in the time available.

Background

The legacy stack couldn’t carry enterprise—or the next platform leap

As GWI scaled into enterprise, user management—grown across multiple codebases—began to show its limits.

Built for simpler organisational models, it struggled with complex contracts, structures, and access rules. Nobody intended fragility; the system hadn’t been designed as a single coherent layer.

There was limited shared understanding of how it behaved end-to-end, or which teams carried which operational workflows day to day.

I led foundational discovery to establish a collective view of reality and inform decisions that could survive API-first growth and enterprise complexity.

Defining the model

OOUX—and a deliberate platform ecosystem

I ran cross-functional OOUX workshops to map objects, relationships, and responsibilities. That shifted the conversation from isolated features and screens to a coherent model we could iterate on together: organisations, membership, roles, access, and entitlements.

User management and API usage weren’t isolated problems—they sit inside a wider platform ecosystem. Alongside core product work, I collaborated with engineers and external vendors to understand where we could leverage existing solutions versus build internally.

WorkOS is confirmed for authentication, identity, sign-in, and user lifecycle.

For API metering and usage tracking we’re still comparing vendors—Metronome and Orb are among the options—and nothing there is signed yet.

That helped us:

  • Spend less time rebuilding commodity plumbing so we could concentrate on problems only GWI can solve
  • Put design effort into the messy middle—orgs, roles, access, and what customers are entitled to
  • Keep the roadmap honest for API-heavy and agent-assisted use cases without multiplying one-off internal systems

Boundaries, not box-ticking. The work clarified what the platform should own end to end versus what should be delegated to external services—so trade-offs in the alpha (and beyond) could be explicit.

Reflections & next steps

What we shipped for alpha, and what we parked on purpose

Alpha had to ship in June. That narrowed the aperture: rather than reinvent identity or org modelling, we had to compose on top of the organisation structures and user records already in our admin system.

Constraint, not a shortcut. Any new experience had to respect those existing trees and user objects. The design job was to make self-serve admin feel inevitable within guardrails we did not own end to end.

With that frame, the trade-off was explicit: deliver clear value fast, and name the bigger system build as a later chapter.

We prioritised

  • Self-serve user management (add, remove, and swap users without chasing support)
  • Enough visibility into access and usage for admins to run a controlled rollout
  • Admin-only controls where actions had to stay gated

We deferred

  • Full organisation hierarchy modelling bespoke to every client shape
  • A complete entitlement system covering every product edge case
  • Deep lifecycle automation spanning the whole user journey

Net result: the alpha stayed useful for the agentic launch, without pretending the entitlement and org layers were solved in one sprint.

The path continues

The outcome (so far)

The work is ongoing, but we’re aligned on the problems, the cost of the status quo, and what “good” looks like next.

  • Stronger steer toward self-serve and a scalable access story
  • Shared groundwork for user management as APIs and automation grow
  • Hands-on contribution to the first self-serve MVP around Agent Octopus
What’s next

New org model, clearer entitlements, WorkOS wired in

Next up is deliberately simple: ship a refreshed organisation structure so accounts match how enterprise customers actually work, get collective agreement on who gets what product access (entitlements), and plug those decisions into our WorkOS integration so sign-in and user lifecycle behave consistently.

Still open. Metering for APIs is on the roadmap; Metronome, Orb, and others stay in the mix until we pick a path—separate from the WorkOS decision.
